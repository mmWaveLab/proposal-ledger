# Lab Material Proposal Skill

用于在本仓库中新增或维护实验室材料申报项目，目标是生成可以直接交给导师审阅、原则上无需再由用户二次改写的 `README.md` 和 `物品名称-申报书.docx`。

## 适用场景

- 用户给出一个拟采购设备、淘宝链接、搜索关键词或研究用途。
- 需要把一次申报归档到 `applications/YYYY/QN/物品名称/`。
- 需要从官方资料、淘宝桌面版价格截图和用户研究背景中形成采购论证。

## 标准流程

1. 建目录：按年份和季度归档，文件夹名使用简短物品名称，可用中文 UTF-8，避免 `/ : * ? " < > |`。
2. 查资料：优先确认官方产品页、官方文档、配件兼容页、规格书、原理图或硬件资料页。
3. 抓价格：用淘宝桌面版短关键词搜索，记录店铺、规格、价格、付款人数和截图；不要把宽泛搜索词写进最终证据。
4. 写 README：包含图片文案资料、资料提取结论、申报成功情况、价格情况、采购理由、使用计划、验收标准。
5. 写申报书源文件：优先维护 `申报书.md`，方便 Git diff 和审查；二级标题只能是 `一、需求分析`、`二、目的用途`、`三、实现目标`、`经费数量`、`四、取得效果或结果`、`五、其他`。
6. 生成 DOCX：运行 `python3 scripts/generate_proposal_docx.py 项目目录`，默认输出到 `exports/proposal-docx/YYYY/QN/物品名称/物品名称-申报书.docx`；DOCX 是生成产物，不提交到 Git。
7. 放图片：在 `申报书.md` 中用 Markdown 图片语法引用关键商品图或官方图，例如 `![商品截图](assets/example.png)`；README 的 `assets/` 中保存官方图、淘宝价格图、后续订单/到货/验收图。
8. 统一字体：生成脚本会将正文中文设为宋体，标题设为黑体，英文和数字设为 Times New Roman，避免依赖非 Windows 常见字体。
9. 更新统计：运行 `scripts/update_stats.py`，确保主 README 的总额、季度、排名和状态自动更新。
10. 质检：运行 `scripts/validate_ledger.py` 和 `scripts/review_proposals.py`；前者检查目录层级、必填字段、价格表、`申报书.md`、本地链接和 DOCX 结构，后者检查申报书中是否残留报销流程话术、内部素材备注、待补充痕迹和过于基础的调试目标。能渲染时用 documents skill 的 `render_docx.py` 生成页面图并目视检查。

## 导师交付门槛

`申报书.md` 和生成的 `物品名称-申报书.docx` 必须按“可直接交给导师”的标准完成，而不是只保存草稿：

- 标题和项目名称要正式、清楚、面向老师理解；避免只用商品型号堆砌标题。
- 摘要必须交代购买对象、选定规格、用途、经费情况和预期结果；如果价格待复核，必须在摘要和经费表中明示，不能伪装成已确认预算。
- 需求分析必须说明“为什么现在需要买”，并连接现有实验问题、数据采集瓶颈、设备联调或研究计划。
- 目的用途必须把关键参数转成实验价值，例如双网口对应网络隔离，M.2 对应本地缓存，RISC-V MCU 对应实时控制，FPGA 对应高速时序采集。
- DOCX 严禁新增模板外一级章节，例如“摘要”“价格与图片凭证”“附件”“验收标准”等都不能作为独立一级标题；商品截图、价格截图、材料说明和完整明细表应并入 `五、其他`。
- 经费数量必须有清晰表格，包含数量、单价、小计、合计、价格来源和价格状态；不得把起售价、低配价或搜索页最低价写成选定规格价。
- 多个小额元器件、耗材或工具以购物车/清单形式申报时，`物品名称-申报书.docx` 必须做到单文件可交付：正文经费表可以汇总为“一批”，但完整明细必须内置在 DOCX 最后的“附件：采购明细表”中，包含名称/型号/品牌/封装/数量/单价/小计等字段；不得要求导师另行打开 CSV/XLS 才能看完整清单，也不得在 DOCX 中提到 CSV/XLS、仓库、归档备份、源文件、README、assets 等外围工程字眼。
- 单独追加的耗材或配件金额也必须进入经费表或附件汇总表，例如“另计某器件 55.00 元、本项目总额 847.70 元”应写成表格行和合计行，不要在正文或附件说明中用散句表达。
- 取得效果或结果必须写成可验收结果，例如完成一次联调、形成采集流程、验证缓存/触发/同步链路，而不是泛泛写“提升研究能力”。
- 其他部分必须保持正式申报语气，可以说明采购对象、材料依据、选定规格和价格信息；不要写“实际采购金额以下单/发票为准”“采购后补充截图”等内部流程话术。
- 不要评价预算金额或经费规模高低，例如“预算金额较低”“金额不高”“价格较低”“经费较少”等都不写；经费信息放在表格中即可。
- 不要虚构或推测实验室管理流程、导师后续要求、到货记录、入库方式、设备编号、使用登记、验收照片、团队长期保留等未由用户明确提供的信息。未确认的信息不要写进 DOCX。
- 文风要正式克制，避免口语、营销话术、夸大承诺和未经证据支持的性能结论。
- 尽量不要把采购目的写成个人“学习”；导师更关心实验室团队级别的实际成果，应优先写成团队可复用平台、样机、数据集、采集流程、测试记录、专利/论文支撑、实验室能力建设等可交付结果。
- 图片应服务于论证，优先放商品外观、接口/结构、价格截图或系统连接图；图片标题写成“商品截图”“价格截图”“材料图”或具体图名，不写“图片凭证”；不放与本次采购无关的配件图片。
- 最终 DOCX 中不得保留“待写”“随便补”“TODO”等草稿痕迹；允许保留“待提交”“待复核价格”，但必须说明待复核原因和后续动作。

## 写作原则

- 申报名称要面向老师理解，优先写“系统能力”而不是商品堆砌。
- 论证要解释为什么“现在需要买”，不要只罗列参数。
- 价格表必须能被 `scripts/update_stats.py` 解析，列名保留 `数量`、`单价(CNY)`、`小计(CNY)`。
- 涉及耗材、配件和工具混合采购时，要区分材料属性；例如工程材料打印板属于可重复使用的非消耗材料，液体胶属于打印辅助消耗材料，不要把二者笼统写成同一种耗材。
- README 可以保存 `assets/*.csv` 或 `assets/*.xls` 作为机器可读归档证据，但面向导师交付的完整清单应以内置 DOCX 附件表为准。`申报书.md` 和 DOCX 绝不能写“CSV/XLS 文件仅作为仓库归档备份”“完整清单见 assets/*.csv”“下载 xls”“仓库归档明细”等外围说明；应改成“完整明细已内置在本申报书中，供审核采购构成和预算明细”。
- 不确定价格必须写 `待复核` 或 `待确认`，不要把起售价、低配价格、搜索页最低价写成选定规格价格。
- 淘宝、商城或订单截图中的商品名、规格和价格可以通过图片识别/OCR提取，但“截图识别”“采购后补充订单截图/到货照片”等证据归档话术只写在 README 或 `assets/README.md`；`物品名称-申报书.docx` 面向导师，应写成正式经费表述，例如“预算单价”“申报金额”“合计”，不写“实际采购金额”“订单及发票为准”等工作流说明。
- 商品截图只用于说明采购对象、规格、价格和材料形态；不要追加“到货后按实验室工具管理要求记录规格、数量、外观和附件完整性”“后续可按导师要求补充到货照片”等未确认流程。
- 如果只确认了部分配件价格，保留已确认行，并在主件行和合计行标记待复核，让统计脚本显示“部分确认”。
- 对技术特性要连接到实验用途，例如双网口对应网络隔离，M.2 对应本地缓存，FPGA 对应高速时序采集。
- 未提交项目的成功情况写“待提交”“待提交后补充”，不要写容易被统计脚本识别成未成功的词。
