# Lab Material Proposal Skill

用于在本仓库中新增或维护实验室材料申报项目，目标是生成可以直接交给导师审阅、原则上无需再由用户二次改写的 `README.md` 和 `申报书.docx`。

## 适用场景

- 用户给出一个拟采购设备、淘宝链接、搜索关键词或研究用途。
- 需要把一次申报归档到 `applications/YYYY/QN/物品名称/`。
- 需要从官方资料、淘宝桌面版价格截图和用户研究背景中形成采购论证。

## 标准流程

1. 建目录：按年份和季度归档，文件夹名使用简短物品名称，可用中文 UTF-8，避免 `/ : * ? " < > |`。
2. 查资料：优先确认官方产品页、官方文档、配件兼容页、规格书、原理图或硬件资料页。
3. 抓价格：用淘宝桌面版短关键词搜索，记录店铺、规格、价格、付款人数和截图；不要把宽泛搜索词写进最终证据。
4. 写 README：包含图片文案资料、资料提取结论、申报成功情况、价格情况、采购理由、使用计划、验收标准。
5. 写 DOCX：严格沿用 `templates/lab-materials/申报书.docx` 的提纲，包括需求分析、目的用途、实现目标、经费数量、取得效果或结果、其他。
6. 放图片：DOCX 中应插入关键商品图或官方图，README 的 `assets/` 中保存官方图、淘宝价格图、后续订单/到货/验收图。
7. 统一字体：正文中文用宋体，标题用黑体，英文和数字用 Times New Roman；正文建议 11pt，避免依赖非 Windows 常见字体。
8. 更新统计：运行 `scripts/update_stats.py`，确保主 README 的总额、季度、排名和状态自动更新。
9. 质检：运行 `scripts/validate_ledger.py`；它会检查目录层级、必填字段、价格表、本地链接和 DOCX 结构。能渲染时用 documents skill 的 `render_docx.py` 生成页面图并目视检查。

## 导师交付门槛

`申报书.docx` 必须按“可直接交给导师”的标准完成，而不是只保存草稿：

- 标题和项目名称要正式、清楚、面向老师理解；避免只用商品型号堆砌标题。
- 摘要必须交代购买对象、选定规格、用途、经费情况和预期结果；如果价格待复核，必须在摘要和经费表中明示，不能伪装成已确认预算。
- 需求分析必须说明“为什么现在需要买”，并连接现有实验问题、数据采集瓶颈、设备联调或研究计划。
- 目的用途必须把关键参数转成实验价值，例如双网口对应网络隔离，M.2 对应本地缓存，RISC-V MCU 对应实时控制，FPGA 对应高速时序采集。
- 经费数量必须有清晰表格，包含数量、单价、小计、合计、价格来源和价格状态；不得把起售价、低配价或搜索页最低价写成选定规格价。
- 取得效果或结果必须写成可验收结果，例如完成一次联调、形成采集流程、验证缓存/触发/同步链路，而不是泛泛写“提升研究能力”。
- 其他部分必须写明价格有效性、订单/发票以下单为准、到货验收和后续补充记录安排。
- 文风要正式克制，避免口语、营销话术、夸大承诺和未经证据支持的性能结论。
- 图片应服务于论证，优先放商品外观、接口/结构、价格截图或系统连接图；不放与本次采购无关的配件图片。
- 最终 DOCX 中不得保留“待写”“随便补”“TODO”等草稿痕迹；允许保留“待提交”“待复核价格”，但必须说明待复核原因和后续动作。

## 写作原则

- 申报名称要面向老师理解，优先写“系统能力”而不是商品堆砌。
- 论证要解释为什么“现在需要买”，不要只罗列参数。
- 价格表必须能被 `scripts/update_stats.py` 解析，列名保留 `数量`、`单价(CNY)`、`小计(CNY)`。
- 不确定价格必须写 `待复核` 或 `待确认`，不要把起售价、低配价格、搜索页最低价写成选定规格价格。
- 如果只确认了部分配件价格，保留已确认行，并在主件行和合计行标记待复核，让统计脚本显示“部分确认”。
- 对技术特性要连接到实验用途，例如双网口对应网络隔离，M.2 对应本地缓存，FPGA 对应高速时序采集。
- 未提交项目的成功情况写“待提交”“待提交后补充”，不要写容易被统计脚本识别成未成功的词。
