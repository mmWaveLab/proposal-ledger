# Lab Proposal Review

用于审查本仓库中的实验室材料申报项目，目标是让 `物品名称-申报书.docx` 成为可以直接交给导师的正式申报材料，而不是内部工作记录。

## 快速流程

1. 运行硬规则审查：

   ```bash
   python3 scripts/review_proposals.py
   ```

2. 对每个 ERROR 必须修改 DOCX；WARNING 需要人工判断，能改成正式表述就改。
3. 修改后继续运行：

   ```bash
   python3 scripts/update_stats.py
   python3 scripts/validate_ledger.py
   python3 scripts/review_proposals.py
   ```

## DOCX 审查原则

正式申报书只写导师需要审批和理解的内容：

- 写采购对象、研究需求、目的用途、实现目标、经费数量、取得效果。
- 写可验收成果，例如完成样机、形成采集链路、完成同步采集、形成数据集、建立实验室工具能力。
- 经费表写“预算单价”“申报金额”“合计”等正式表述。

以下内容不应出现在 DOCX 正文：

- 报销/下单流程话术：`本次经费按当前记录价格测算`、`实际采购金额`、`以下单页为准`、`订单及发票为准`、`发票为准`。
- 内部素材备注：`截图识别`、`接口标注`、`待补充`、`采购后补充订单截图`、`到货照片补充到 assets`。
- 不准确的证据表述：`图片凭证`、`凭证材料`、`价格凭证`。商品图、淘宝截图和价格截图只能写成“商品截图”“价格截图”“材料图”“材料说明”，不要写成正式凭证。
- 入门调试目标：`点灯`、`建立工程`、`烧录测试`、`系统启动`、`基础 Python`。
- 个人学习导向：把“学习”“练习”“熟悉工具链”改成实验室成果、样机、流程、数据或工具能力。

## 推荐替代表述

- 不写：`本次经费按当前记录价格测算，实际采购金额以审批后订单及发票为准。`
- 可写：`本项目预算用于购置表中列明的设备和配套材料，形成可用于实验室项目的硬件条件。`

- 不写：`接口标注图用于说明双网口和 M.2。`
- 可写：`材料图显示该设备具备双千兆网口、M.2 存储扩展和板载接口条件，可支撑采集终端部署。`

- 不写：`图片凭证：`
- 可写：`商品截图：` 或 `价格截图和商品材料图：`

- 不写：`完成烧录、点灯、建立工程。`
- 可写：`完成采集链路联调，形成接口映射、数据流路径、样例数据和实验记录。`

## README 与 DOCX 分工

- README 可以保存淘宝截图、订单截图、到货照片、价格来源、复盘记录、后续补充事项。
- DOCX 只放正式申报内容；图片可以放商品图、结构图、价格材料，但图片标题和正文不要写成内部素材管理语气。
